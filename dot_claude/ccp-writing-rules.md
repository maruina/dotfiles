# PLAN.md Writing Rules

These rules govern the structure and content of PLAN.md files. Both `/ccp-plan` (creation) and `/ccp-review` (reconciliation) must follow them.

## Writing Rules

- **One-sentence problem statement** — the Jira ticket link provides full context. The problem section is a single sentence orienting the reader, nothing more.
- **Approach section is critical** — this is where the reviewer gets value. Detail the approach enough for the reviewer to give meaningful feedback on the direction.
- **Open Questions right after Approach** — the reviewer needs to weigh in on design decisions before seeing the artifact. Present options with trade-offs and a recommendation for each.
- **Show the core design artifact** — include one code snippet for the central design decision (the CRD schema, key interface, primary struct). Place it in the "Key Design" section after Open Questions. This is the part reviewers need to see to give feedback.
- **No other inline code** — everything outside the core artifact is described in prose. No boilerplate code (scheme registration, helper functions, codegen output).
- **No file lists** — "Files to Create" and "Files to Modify" are implementation details for `/ccp-implement`. The plan reviewer cares about *what* and *why*, not which files to touch.
- **No boilerplate** — skip scheme registration, BUILD.bazel contents, codegen markers, snapshot commands, and similar mechanical details. These are implementation-phase concerns.
- **No Steps section** — do not add a `## Steps`, `## Implementation Steps`, or any numbered implementation sequence. The plan describes WHAT to build and WHY. `/ccp-implement` handles HOW.
- **No implementation details** — codegen steps, snapshot commands, build verification steps, specific command sequences, directory structure setup, and file scaffolding belong to `/ccp-implement`, not the plan.
- **No "future plan" sections** — stay focused on current scope. No "First Absorption Candidates", context for future tickets, or roadmap items.
- **No "Review Notes" section** — the updated plan stands on its own after reconciliation.
- **Gazelle awareness** — BUILD/bzl files for Go are generated by Gazelle. Never propose writing them manually. Note this in the plan if relevant.

## PLAN.md Template

```markdown
# Implementation Plan: <title>

**Source**: <Jira ticket link or "free text">
**Date**: <YYYY-MM-DD>
**Author**: @<github-username>

## Problem
<one sentence — the Jira ticket has the full context>

## Approach
<chosen approach with rationale — why this over alternatives. This is the most important section.>

## Open Questions
- <question> — **Option A**: <pros/cons>. **Option B**: <pros/cons>. Recommendation: <which and why>.

## Key Design
<code snippet of the core artifact: the CRD schema, key interface, or primary struct. Only the central design decision that reviewers need to evaluate. No boilerplate.>

## Testing Strategy
- <specific test cases, not generic "add tests">
- Reference existing test patterns from research

## Risks & Mitigations
- <concrete risks, not hypotheticals>

## Out of Scope
- <explicitly excluded items>
```

## Section Order

Problem → Approach → Open Questions → Key Design → Testing Strategy → Risks & Mitigations → Out of Scope
